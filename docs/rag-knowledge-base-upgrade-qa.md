# RAG 知识库升级方案 Q&A

> 整理自产品设计讨论，适用于客服/行业知识库场景下的 RAG 系统升级决策。

---

## 一、当前系统的问题

### Q：当前系统是怎么处理文档的？

当前逻辑：对每篇文档，不论长短，直接截断到固定长度（1500 字符），然后对整段文字进行向量化。查询时通过分词（BM25）或向量相似度搜索匹配。

**问题：**
- 截断会破坏语义，长文档的后半段信息完全丢失
- 全文向量化导致向量"语义太宽泛"，检索精度低
- 短文档和长文档用同一套逻辑，命中率参差不齐

---

## 二、分块策略

### Q：更好的分块方式是什么？

参考 `llm_wiki` 项目的实现，应使用**标题感知的递归分块器**：

1. 按 `##` 标题优先切分
2. 每块目标大小 1000 字符，最大 1500 字符
3. 相邻块之间保留 200 字符的重叠（overlap），防止语义断裂
4. 代码块、表格不切分（保持完整性）
5. 摄入前先剥离 YAML frontmatter

这样每个块在语义上是完整的，向量化后检索精度更高。

### Q：对于特定行业文档，这种分块策略是否会减弱效果？

会的，行业文档的问题在于：

- 一段话包含多个独立事实（规格参数、条件限制混合在一起）
- 按标题切分可能仍然粒度太粗
- 术语密度高，通用分块器不理解业务边界

这时候需要在分块之上，叠加**命题提取（Proposition Extraction）**。

## 三、命题提取

### Q：什么是命题提取？

让 LLM 阅读原文，从中提取出**原子事实**，每条命题满足：

- 独立可验证（不依赖上下文就能理解）
- 只表达一个事实
- 保留关键上下文（主体、条件、数字等）

**示例：**

原文：
> 本产品额定电压 380V±10%，环境温度不超过 40°C，S1 级客户享受免费上门服务。

提取后：
- 产品额定工作电压为 380V，允许偏差 ±10%
- 产品最高工作环境温度为 40°C
- S1 级客户享受免费上门维修服务

**向量化的是这些命题，而不是原文。**

### Q：命题提取是否适合特定行业知识库？

非常适合。行业知识库的核心挑战是"一段文字包含多个条件和限制"，命题提取能把这些条件拆分成独立检索单元，大幅提升召回的准确性。

## 四、提示词体系

### Q：命题提取需要哪些提示词？

完整的提示词体系由三部分组成：

#### 1. 提取提示词（Extraction Prompt）
- 作用：离线摄入阶段，教 LLM 如何阅读文档、抽取原子命题
- 决定了知识库的质量天花板
- 包含：提取规则、命题格式要求、哪些内容不需要提取

#### 2. 回答提示词（Answer Prompt）
- 作用：在线查询阶段，教 LLM 拿到检索出的命题后如何组织回答
- 包含：回答格式、语气风格、引用方式、不确定时如何表达

#### 3. 示例库（Few-shot Example Library）
- 作用：给 LLM 看"好的命题长什么样"，弥补规则描述的不精确性
- 结构：原文片段 → 正面示例（应提取的命题）+ 反面示例（不应提取的）
- 随业务积累动态增长，但每次调用只注入 3-5 条最相关的

### Q：为什么示例库只需要几十条，就能覆盖庞大的知识库？

**示例库覆盖的不是知识内容，而是提取模式。**

一家公司的文档类型通常 10-20 种（产品规格、故障排查、退换货政策、价格说明……），每种类型给 2-3 个示例，共 30-60 条，LLM 就能学会"这类文档应该怎么切"。

类比：教一个编辑"怎么写摘要"，不需要把所有文章的摘要都给他看，给几篇不同类型的示范就够了。

**示例库会失效的情况：**
- 边界情况未覆盖（如嵌套条件、例外规则）
- 术语多义/别名问题（"退款" vs "退货退款"）
- 复杂表格结构

遇到这些情况，针对性地补充示例或增加专项规则。

## 五、提示词套件路由系统

### Q：如何管理不同文档使用不同的提示词？

将三类提示词打包成一个**套件（Prompt Set）**，每个套件对应一类文档：

```
PromptSet {
  id, name           // 如："退换货政策类文档"
  extractionPrompt   // 摄入时使用
  answerPrompt       // 查询时使用
  examples[]         // 示例库
  routingRules[]     // 自动路由规则
  isDefault          // 是否为兜底套件
}
```

### Q：自动路由规则怎么设计？

按优先级叠加，越靠前越快、越精确：

| 优先级 | 规则类型 | 示例 | 说明 |
|--------|----------|------|------|
| 1 | 路径/文件夹 | `/knowledge/refund/**` → 退换货套件 | 最精确，无 LLM 开销 |
| 2 | 标题关键词 | 标题含"故障/报错/无法" → 故障排查套件 | 中等精度，配置简单 |
| 3 | 内容分类 | 取前 500 字 → LLM 分类 → 返回套件类型 | 兜底，有额外 LLM 开销 |
| 4 | Default 套件 | 以上都未匹配 | 通用处理 |

**实践建议：** 早期只做规则 1 + 2，未匹配的走 Default 套件并人工审查，积累经验后再补规则。不要一开始就依赖 LLM 内容分类。

### Q：维护界面需要哪些功能？

**套件管理页**
- 创建/编辑套件，填写三类提示词
- 上传示例（原文片段 → 正例 / 反例）
- 配置路由规则

**路由测试页**（重要，容易被忽略）
- 输入文档标题/路径，模拟路由结果，确认选了哪个套件
- 输入完整文档片段，预览提取出的命题质量

**摄入覆盖页**
- 查看已摄入文档各自用了哪个套件
- 支持单篇手动指定套件，触发重新提取

## 六、实施路线

### Q：完整升级需要哪些步骤？

**阶段一：分块升级（基础，必做）**
- 将当前截断逻辑替换为标题感知的递归分块器
- 参考 `llm_wiki/src/lib/text-chunker.ts` 实现，移植到 Spring Boot 后端

**阶段二：命题提取管道（核心）**
- 增加摄入管道中的 LLM 提取步骤：原文块 → 命题列表
- 命题单独存储、单独向量化

**阶段三：提示词套件管理（产品化）**
- 数据库表：`prompt_set`、`prompt_example`、`routing_rule`
- 后端 API：套件 CRUD、路由规则配置
- 前端维护页：供客服人员维护套件和示例

**阶段四：路由自动化（成熟期）**
- 路径 + 关键词规则引擎
- 可选：LLM 内容分类作为兜底

### Q：不同规模的知识库，方案选择有何差异？

| 规模 | 建议方案 |
|------|----------|
| 小型（< 1 万文档） | 分块升级 + 单一通用提示词套件，几十条示例够用 |
| 中型客服知识库 | 分块 + 命题提取 + 多套件路由，按文档类型维护示例 |
| 超大型/专业领域 | 上述全套 + 考虑微调专用提取模型替代 few-shot |
